Getting Started

When should you use aria?

A great starting point for expanding your accessibility skills is to learn{' '} when and how to use aria.

Usually all of the problems are caused by us the developer, these may include:

  • Accessibility
  • Responsiveness at different screen sizes
  • Sites that can't be used without Javascript

All of the HTML elements that are available to a developer come with specific roles (and other features) baked into them. A div element does not naturally come with an onClick, and it wouldn't normally be focusable.

Through the use of Javascript we tend to 'fix' these problems (because we can) and move on.

General Rules

1 Complexity can be confusing

When / if you write complex logic with Javascript, hide things with CSS, add extra abilities to HTML elements (i.e. add focusability to a div) it will probably be appropriate to add an aria-role to clarify the intent of the component.

ARIA, by design, has no effect on how elements are displayed or behave in browsers. It does not add new functionality and is meant to act only as an extra descriptive layer for screen readers

2 Semantic / Correct HTML conveys how to use the website properly.

A website without any CSS or javascript is probably going to be 99% responsive, and if we made sure to only use Semantic markup it would probably be Accessible to 99.99% of people.

  • Appropriate Heading elements (h1, h2, h3 etc.)
  • Landmarks to identify where things are (heading, main, footer)
  • topics / subjects that are self contained and wrapped with-in an article, with a heading, and separated into sections.
